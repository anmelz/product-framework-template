Getting Started

Todo lo que el usuario necesita hacer antes de abrir la primera conversación con Claude. Sigue el flujo correspondiente según si el proyecto es nuevo o ya está iniciado.

Paso 1 · Crea la carpeta local del proyecto
☐
Crear carpeta con estructura Cliente/Proyecto
Crea la carpeta siguiendo la estructura ~/Projects/NombreCliente/NombreProyecto. Ejemplo: ~/Projects/AcmeCorp/DashboardMVP. Esta separación permite tener varios proyectos del mismo cliente organizados bajo la misma carpeta padre.
Paso 2 · Code Clona el template en esa carpeta
☐
Abrir Code apuntando a la carpeta del proyecto
Abre la app de Claude → tab Code → selecciona la carpeta creada en el Paso 1. Luego envía este mensaje:
Mensaje para Code
Setup inicial del proyecto

Clona el template del framework en esta carpeta:
git clone https://github.com/anmelz/product-framework-template .

Luego crea un repositorio en GitHub llamado [nombre-proyecto], conéctalo como origin y haz el push inicial.

Code clona el template con toda la estructura (.claude/, _context/, docs/) y configura el repo en GitHub. Desde este momento Code ya tiene el contexto del framework automáticamente.

Paso 3 · Cowork Crea el proyecto y pega las instrucciones
☐
Cowork → New Project → seleccionar la carpeta del proyecto
Crea el proyecto en Cowork apuntando a la carpeta local del repositorio.
☐
Pegar el contenido de _context/framework.md en las instrucciones del proyecto
Al crear el proyecto en Cowork hay una sección de instrucciones — pega ahí el contenido de _context/framework.md. Esto le da a Cowork el contexto del framework para todas las sesiones de este proyecto. Cada miembro del equipo repite este paso al crear su propia copia local del proyecto en Cowork.
Paso 4 · Arranca la Fase 1 en Cowork

Con todo configurado, abre el proyecto en Cowork y envía el contexto inicial del cliente para comenzar la Fase 1 · Investigación.

El Project Audit se hace en Cowork. En vez de copiar y pegar contexto en el prompt, adjunta los archivos directamente — Cowork lee .md y .html sin problema. Organiza los archivos antes de empezar.

Fases del proceso
-1
🎯
Calificación
¿Vale la pena invertir en este prospecto? · 30 min máximo
claude-sonnet-5
Actividades
  • UsuarioContexto rápido del prospecto: quién es, qué quiere, cómo llegó.
  • ClaudeEvaluación BANT adaptada: Budget aproximado, Autoridad de decisión, Necesidad real, Timing.
  • ClaudeBúsqueda rápida del prospecto: tamaño, industria, señales de capacidad de pago.
  • ClaudeScore go/no-go con recomendación de track (Fast o Full).
  • UsuarioDecisión final: proceder, descartar, o nutrir para más adelante.
Output
🎯 Score de calificación 🛤️ Track recomendado

Si es no-go, el prospecto se descarta o se guarda para nutrir más adelante — sin invertir más. Si es go, se define el track y se procede al setup.

Criterios de calificación — mínimo 3 de 4 para proceder
  • Budget: hay indicios de capacidad y disposición de inversión acorde al proyecto
  • Autoridad: el contacto puede decidir o tiene acceso directo a quien decide
  • Necesidad: el problema es real, actual y le cuesta dinero o tiempo al prospecto
  • Timing: hay urgencia o un evento que motiva resolver esto ahora
01
🔍
Investigación
Recopilación de contexto e investigación online profunda
claude-fable-5
Actividades
  • UsuarioDump completo de contexto: idea, cliente, oportunidad, problema, situación.
  • UsuarioConfigurar herramientas y repositorios del proyecto (GitHub repo, carpeta docs/).
  • ClaudeInvestigación online profunda sobre el cliente, mercado inicial y competencia — con agentes `investigador` en paralelo (uno por objetivo: competidor, mercado, prospecto).
  • ClaudeCrear el artifact inicial y compilar toda la información encontrada.
  • ClaudeGenerar lista de preguntas y gaps pendientes.
Output
📄 Sección de Investigación en Artifact ❓ Lista de preguntas y gaps pendientes 🛠️ Setup de herramientas completado

El artifact se crea en esta fase. La lista de gaps guía la siguiente fase de Discovery.

Criterios de salida
  • Contexto inicial completo recibido del usuario
  • Herramientas y repositorios configurados
  • Investigación online completada
  • Lista de preguntas pendientes generada y documentada en el artifact
02
💡
Discovery & Definición
Definición completa del producto y la visión
claude-fable-5
Actividades
  • ClaudeRevisión y resolución de la lista de preguntas pendientes de la fase anterior.
  • ClaudePreguntas estratégicas adicionales según el contexto del proyecto.
  • UsuarioResponder preguntas y agregar contexto libre en cualquier momento.
  • ClaudeSintetizar visión, objetivos, scope y funcionalidades clave.
  • UsuarioAprobar la visión y el scope antes de avanzar.
Output
🎯 Visión del producto 📋 Objetivos y KPIs 📐 Scope definido ⚙️ Funcionalidades clave 🗺️ Estructura general

El proyecto queda completamente definido "en papel". Todo se documenta en el artifact.

Criterios de salida
  • Todas las preguntas pendientes de investigación resueltas
  • Visión y propósito del producto claramente definidos
  • Scope, funcionalidades y estado final aprobado por el usuario
  • Sección de Discovery completada en el artifact
03
📊
Análisis de Mercado
Viabilidad, competencia y oportunidad de mercado
claude-fable-5
Actividades
  • ClaudeAnálisis profundo de competencia (directa e indirecta).
  • ClaudeEvaluación del tamaño de mercado (TAM/SAM/SOM).
  • ClaudeIdentificación de tendencias, oportunidades y amenazas.
  • ClaudeAnálisis de viabilidad del producto.
  • UsuarioComplementar con datos propios o conocimiento interno (opcional).
  • UsuarioAprobar el análisis antes de avanzar al Gate.
Output
🏆 Análisis de competidores 📈 Tamaño de mercado 🚀 Oportunidades ⚠️ Riesgos y amenazas 🎯 Posicionamiento

Documento de análisis de mercado agregado al artifact. Base para la propuesta al cliente.

Criterios de salida
  • Análisis de competencia completado
  • Viabilidad del producto confirmada
  • Estrategia de posicionamiento definida
  • Análisis documentado y aprobado en el artifact
🤝
Gate · Presentación al Cliente
Propuesta + Prototipo — la combinación que cierra
Prototipo ≠ MVP: el prototipo es la muestra visual clickeable de cómo se vería el producto (sin funcionalidad real) — herramienta de venta pre-firma. El MVP (Fase 6) sí es funcional. ¿La propuesta lleva prototipo? Claude propone y el usuario confirma: el default es CON prototipo — un cliente que toca su producto cierra a una tasa dramáticamente mayor, y con IA cuesta horas. La excepción (sin prototipo, justificada): deals muy chicos, prospectos ya decididos, o discovery demasiado débil para anclar el prototipo en la realidad del cliente. El prototipo no tiene tope fijo de pantallas — entre más completo y con datos reales del prospecto, más fuerte. Si crece bastante más allá de lo planeado en Discovery, pasa por Scope Creep antes de seguir.
✓ Cliente aprueba
Se avanza a Fase 4 · Planificación Técnica. El feedback del cliente se documenta en el artifact.
↩ Cliente rechaza o pide cambios
Se regresa a Fase 2 · Discovery con el feedback documentado. Claude incorpora los cambios y el proceso avanza nuevamente.
04
⚙️
Planificación Técnica
Arquitectura, stack y roadmap de desarrollo
claude-opus-4-8
Actividades
  • ClaudeEvaluar todo el contexto acumulado y proponer el stack óptimo.
  • ClaudeDiseñar arquitectura del sistema (frontend, backend, DB, infra).
  • ClaudeDefinir integraciones y servicios necesarios.
  • ClaudeGenerar estimaciones y el contenido del CLAUDE.md.
  • UsuarioRevisar, ajustar y aprobar la propuesta técnica.
  • AmbosCommitear CLAUDE.md al repo (Claude Code si tiene acceso git, usuario si no).
Output
🛠️ Stack tecnológico 🏗️ Arquitectura del sistema 🔌 Integraciones ⏱️ Estimaciones 🗓️ Roadmap técnico

Propuesta técnica completa. El stack se decide con el panorama completo del proyecto.

Criterios de salida
  • Stack tecnológico definido y aprobado
  • Arquitectura del sistema documentada
  • Estrategia de entornos definida: local / staging / production, con DBs separadas y secrets por entorno (sección "Entornos" del CLAUDE.md)
  • Perfil de seguridad definido: sensibilidad de datos, modelo de auth, superficie de ataque y requisitos del producto (sección "Seguridad" del CLAUDE.md)
  • Estimaciones aceptadas por el equipo y/o cliente
  • Propuesta técnica agregada al artifact
05
🎨
Diseño / UX
Wireframes, flujos, diseño visual y design system
claude-opus-4-8
Actividades
  • UsuarioDar requirements de diseño: preferencias, referencias, restricciones de marca (opcional).
  • AmbosDefinir el nivel de interactividad: Claude propone según industria/producto (muy dinámico · moderado · sobrio) con justificación, el usuario confirma. Default del framework: dinámico e interactivo.
  • ClaudeDefinir flujos de usuario y arquitectura de información.
  • ClaudeCrear wireframes de todas las pantallas y componentes.
  • ClaudeDiseño visual completo: paleta, tipografía, componentes, estados.
  • ClaudePrototipado interactivo con Claude Design.
  • UsuarioDar feedback, aprobar prototipo y exportar assets desde Claude Design.
  • AmbosHandoff a Code: exportar el output aprobado a docs/design/ del repo — prototype/ (pantallas), design-tokens.md, components.md, flows.md.
Output
🗺️ Flujos de usuario 📐 Wireframes completos 🎨 Design system ⚡ Prototipo interactivo 📋 Specs para desarrollo

El design system y los prototipos se exportan a docs/design/ del repo — el input directo que Claude Code lee al arrancar la fase de MVP, para construir sobre lo aprobado en vez de rediseñar.

Criterios de salida
  • Nivel de interactividad propuesto por Claude y confirmado por el usuario
  • Todos los flujos de usuario definidos y aprobados
  • Wireframes de todas las pantallas completados
  • Design system documentado (colores, tipografía, componentes)
  • Prototipo aprobado por el usuario antes de pasar a código
  • Handoff exportado a docs/design/: prototype/, design-tokens.md, components.md, flows.md
06
🚀
MVP
Producto mínimo viable funcional
claude-sonnet-5 + Code
Actividades
  • ClaudeDefinir scope mínimo viable (con aprobación del usuario).
  • CodeSetup de entornos ANTES de la primera feature: hosting con local / staging / production, DBs separadas por entorno, secrets por entorno, pipeline verificado con un cambio trivial.
  • CodeLeer docs/design/ (handoff de Fase 5) antes de la primera pantalla: tokens al config de estilos, componentes según inventario, pantallas según prototipo.
  • CodeDesarrollo iterativo del MVP construyendo sobre el design system — flujo local → staging. Features independientes en paralelo vía agentes (`implementador`, `tester`); lo compartido (schema, config, layout) serializado.
  • CodeCierre de fase con agentes: `verificador-qa` contrasta los criterios de salida y `revisor-seguridad` (Opus) audita el baseline R9 — ambos reportan antes de aprobar.
  • CodeTesting funcional y corrección de errores.
  • AmbosDeploy a ambiente de pruebas (Claude Code si tiene acceso git/deploy, usuario si no).
  • UsuarioValidar el MVP, dar feedback y aprobarlo antes de continuar.
Output
💻 MVP funcional deployado 🧪 Resultados de testing 💬 Feedback inicial

Producto funcional con las features core. Base para validar la propuesta de valor real.

Criterios de salida
  • Entornos configurados y pipeline local → staging → production verificado
  • MVP completado y deployado en staging
  • Testing básico completado sin errores críticos
  • Baseline de seguridad verificado: validación server-side, AuthZ en flujos core, dependencias sin vulnerabilidades críticas
  • Validado y aprobado por el usuario/cliente
  • Feedback documentado para incorporar en desarrollo completo
07
✅
Desarrollo Completo
Producto final en producción
claude-sonnet-5 + Code
Actividades
  • CodeDesarrollo iterativo de todas las funcionalidades del scope.
  • CodeIncorporación del feedback del MVP.
  • CodeQA completo en staging: edge cases, performance, seguridad, compatibilidad.
  • CodeOptimización de rendimiento.
  • AmbosDeploy a producción — solo tras QA aprobado en staging, con DB y secrets de production propios, migraciones ya verificadas en staging (Claude Code si tiene acceso, usuario si no).
  • UsuarioAprobación final y entrega al cliente.
Output
🏁 Producto completo en producción 📚 Documentación técnica 🔧 Entorno configurado

Producto finalizado, documentado y entregado al cliente.

Criterios de salida
  • Todas las funcionalidades del scope implementadas
  • QA completado sin errores críticos
  • Revisión de seguridad completa aprobada: checklist del perfil de seguridad, headers, rate limiting, RLS/AuthZ, dependencias sin críticas ni altas
  • Producto deployado en producción
  • Cliente ha aprobado el entregable final
08
🔭
Post-Lanzamiento
Monitoreo, mantenimiento e iteración continua
claude-sonnet-5
Actividades
  • UsuarioConfigurar herramientas de monitoreo (analytics, alertas, uptime).
  • ClaudeAnalizar métricas y feedback de usuarios.
  • CodeBug fixes, parches de seguridad y mantenimiento correctivo — siempre por el ciclo local → staging → production.
  • AmbosCumplir el soporte pactado en el SOW: alcance, tiempos de respuesta y duración firmados. Trabajo fuera de ese alcance = nueva propuesta o scope change.
  • AmbosEvaluar si una iteración mayor amerita reiniciar el framework desde Discovery.
  • UsuarioTomar la decisión final sobre iteraciones y roadmap.
Output
📊 Reportes de métricas 🐛 Bugs resueltos 💬 Feedback de usuarios 🔄 Decisión de iteración

Si una iteración es suficientemente grande, puede reiniciar el framework completo desde Fase 2 · Discovery con el contexto acumulado.

Criterios para reiniciar el framework
  • La iteración propuesta afecta la propuesta de valor o el modelo de negocio
  • Requiere cambios estructurales en arquitectura o stack
  • Impacta más del 30% de las funcionalidades existentes
  • Implica un nuevo análisis de mercado o reposicionamiento
Tracks — velocidad según el deal

La velocidad al Gate importa: los prospectos se enfrían. La Fase -1 determina qué track aplicar según el tamaño y complejidad del deal. Ambos tracks producen propuesta + demo — la diferencia es la profundidad.

⚡
Fast Track
Deals pequeños/medianos · Propuesta en 24–48h
1 Fases 1–3 comprimidas en una sola sesión intensiva de Cowork
2 Investigación enfocada: solo el prospecto y 2–3 competidores directos
3 Discovery basado en arquetipo — solo se pregunta lo que difiere del patrón
4 Demo construido con IA en horas, con branding del prospecto
5 Gate en 24–48 horas desde el primer contexto
🔬
Full Track
Deals grandes o complejos · Proceso completo
1 Fases 1, 2 y 3 completas como sesiones separadas
2 Investigación profunda: mercado completo, TAM/SAM/SOM con fuentes
3 Discovery exhaustivo con múltiples iteraciones
4 Demo más elaborado: flujo completo del caso de uso principal
5 Aplica cuando el deal justifica la inversión de profundidad
Arquetipos de proyecto — la verdadera productización

La mayoría de los proyectos encajan en patrones repetibles. Cada arquetipo tiene stack pre-decidido — la Fase 4 se vuelve más rápida y las estimaciones mejoran con cada proyecto entregado. A futuro, un starter kit repo (auth, layout base, dashboard shell, pagos, deploy config) podría hacer que cada MVP arranque mucho más cerca del final en vez de 0% — pendiente de construir una vez haya suficientes proyectos reales para basarlo en patrones concretos, no en suposiciones.

Arquetipo Incluye Stack base Estimación
SaaS MVP Web app con auth, dashboard, billing, landing de marketing Next.js + Tailwind + Postgres + Stripe + Vercel [calibrar]
Marketing + Producto Sitio de marketing + web app de producto + backend compartido Next.js + Tailwind + Postgres + Vercel [calibrar]
Plataforma interna Herramienta interna: dashboards, gestión de datos, reportes, roles Next.js + Tailwind + Postgres + auth SSO [calibrar]
App móvil + backend App nativa/híbrida con API, auth y panel de administración React Native / Expo + API + Postgres [calibrar]
Custom No encaja en ningún patrón — proceso completo de Fase 4 Se define en Planificación Técnica Por proyecto

Modelos recomendados por fase

Claude no puede cambiar el modelo por sí mismo — lo selecciona el usuario en la app. Por eso el framework exige que en cada Status Check y en cada cierre de fase ("¿Qué sigue?") Claude indique el modelo recomendado del siguiente paso, y avise explícitamente si hay que cambiarlo en el selector. Disponibles de menor a mayor capacidad: Haiku 4.5 · Sonnet 5 · Opus 4.8 · Fable 5 — con libertad de subir para una tarea puntual compleja o bajar a Haiku para lo trivial, volviendo luego al recomendado.

Modelo Cuándo usarlo Fases
claude-fable-5 Tareas abiertas y sin estructura predefinida que requieren el máximo nivel de razonamiento: investigación profunda sin límites claros, discovery desde cero, análisis de mercado exhaustivo con múltiples fuentes. Ideal para sesiones largas y complejas. Fase 1 (research), 2, 3

Propuesta Comercial — Gate al Cliente

La sección de propuesta para el cliente en el artifact debe ser completa para que el gate sea efectivo. El cliente no puede tomar una decisión informada sin estos componentes.

Componente Descripción
Resumen ejecutivo El problema que se resuelve, por qué importa, y qué hace la solución en 3–5 oraciones. Escrito para un ejecutivo, no para un técnico.
Propuesta de valor Qué diferencia a esta solución de las alternativas existentes. Basada en el análisis de mercado de Fase 3.
Scope y funcionalidades Qué incluye exactamente la solución. Lo que está en scope y explícitamente lo que está fuera de scope.
Timeline estimado Fases, hitos y tiempos estimados. Basado en las estimaciones de Fase 4 si ya se hicieron, o en rangos si es pre-aprobación.
Estructura de pricing Modelo de cobro (proyecto fijo, retainer, por fase) y rangos de inversión. No necesariamente precio exacto en esta etapa.
Próximos pasos Qué pasa si el cliente aprueba: qué se necesita de su parte, cuándo arrancaríamos, primer entregable concreto.
Reglas de enforcement
⚠️
Este framework es obligatorio, no opcional
Claude no puede avanzar a una fase siguiente sin que se cumplan todos los criterios de salida de la fase actual. Si el usuario intenta saltarse el proceso, Claude detiene la ejecución y explica qué falta.
R1
Verificación de criterios antes de avanzar
Antes de comenzar cualquier fase, Claude verifica que todos los criterios de salida de la fase anterior estén cumplidos. Si alguno falta, Claude lo lista explícitamente y no procede.
R2
Redirección activa al proceso
Si el usuario solicita trabajo que corresponde a una fase posterior sin haber completado la actual, Claude lo redirige: explica en qué fase se encuentran, qué falta, y por qué importa no saltarse ese paso.
R3
Override consciente para excepciones
Si el usuario necesita saltarse fases por urgencia u otra razón válida, puede activar un override consciente. Claude lista los riesgos específicos de saltarse ese paso, y el usuario debe confirmar explícitamente escribiendo Confirmo override. El override queda documentado en el artifact.
R4
Input libre del usuario siempre permitido
En cualquier fase, el usuario puede agregar contexto, corregir información o ajustar decisiones tomadas. El input libre no rompe el proceso — lo enriquece. Claude incorpora los cambios y actualiza el artifact.
R5
Artifact como fuente de verdad
Toda decisión, cambio o override queda documentado en el artifact. Si hay contradicción entre lo que el usuario dice en chat y lo que está en el artifact, Claude señala la discrepancia antes de proceder.
R6
Herramienta correcta por fase
Claude debe indicar al usuario si la tarea requiere cambiar de herramienta (ej. pasar de Chat a Cowork para investigación, o a Claude Code para desarrollo). No se ejecuta trabajo complejo de una herramienta desde otra.
R7
Datos client-facing verificados
Todo dato cuantitativo que llegue a la propuesta o al demo (TAM, datos de competidores, métricas de mercado) debe tener fuente citada y verificable, o presentarse explícitamente como estimación. Un dato inventado frente a un prospecto que conoce su industria destruye la credibilidad — Claude nunca presenta números sin respaldo como hechos.
R8
Scope Creep controlado
Cualquier expansión del scope original — venga del cliente, del usuario o del propio proceso (ej. un prototipo del Gate que crece más allá de lo planeado) — pasa por el flujo formal: evaluación de impacto → presentación → confirmación explícita con Confirmo scope change → documentación en PROGRESS.md. Ver la sección "Manejo de Scope Creep" para el flujo completo.
R9
Seguridad por diseño
Todo producto cumple el perfil de seguridad definido en Fase 4 (sensibilidad de datos, auth, superficie de ataque) y el baseline del CLAUDE.md: validación server-side, AuthN+AuthZ en cada endpoint, queries parametrizadas, secrets fuera del código, rate limiting, headers, dependencias auditadas. Se aplica mientras se construye, no como parche al final. Un hallazgo crítico de seguridad bloquea el deploy igual que un error crítico funcional — saltárselo requiere Override Consciente.

Manejo de Scope Creep

Cualquier solicitud que expanda el scope original del proyecto — ya sea del cliente, del usuario, o una decisión del equipo — debe seguir este proceso formal. Nada se agrega sin evaluación de impacto y aprobación explícita.

📥
Propuesta de cambio
El usuario describe el cambio solicitado: qué se quiere agregar, modificar o eliminar del scope original.
→
🔍
Evaluación de impacto
Claude evalúa: qué fases afecta, cuánto esfuerzo adicional implica, qué decisiones anteriores quedan invalidadas, y qué riesgos introduce.
→
✅
Aprobación explícita
El usuario aprueba o rechaza con contexto claro. Si aprueba, confirma entendiendo el impacto completo.
→
📝
Actualización y commit
Claude actualiza las fases afectadas en el artifact, el PROGRESS.md, y hace commit a GitHub con tag scope:change.
Ejemplo de evaluación de impacto
Scope Change Request Evaluado por Claude
Cambio solicitado Agregar módulo de pagos recurrentes
Fases afectadas Fase 2 (scope), Fase 4 (arquitectura + integración Stripe), Fase 5 (nuevas pantallas), Fase 6–7 (implementación)
Decisiones invalidadas Arquitectura de base de datos definida en Fase 4 requiere nuevas tablas. Stack de pagos no contemplado.
Esfuerzo estimado +2–3 semanas de desarrollo adicional
Riesgos Complejidad de compliance (PCI-DSS), nueva dependencia externa, posible retraso en entrega
Para aprobar Escribe "Confirmo scope change" para proceder. Se actualizarán las fases 2, 4, 5 y el CLAUDE.md.
Testing & QA

El testing está integrado en las fases de desarrollo como criterio de salida obligatorio — no como actividad opcional. No se puede avanzar de fase ni hacer deployment con errores críticos sin resolver.

🧪
Fase 6 · Testing del MVP
Testing funcional — criterio de salida obligatorio
1 Testing funcional de todas las features del scope mínimo
2 Casos de uso principales cubiertos y validados
3 Flujos críticos probados de extremo a extremo
4 Errores críticos resueltos — no se cierra la fase con bugs críticos abiertos
5 Resultados documentados en el artifact
🔬
Fase 7 · QA Completo
QA integral — requerido antes del deployment
1 QA funcional completo incluyendo edge cases
2 Testing de performance y carga (si aplica al tipo de producto)
3 Revisión de seguridad básica antes del deployment
4 Compatibilidad cross-browser y cross-device
5 Todos los resultados documentados y aprobados antes de producción
🚫
Regla de testing
Claude no ejecuta ni aprueba deployment a producción si hay errores críticos o si el QA de Fase 7 no está completo. Si el usuario insiste en hacer deployment sin QA completo, se activa el Override Consciente y se documenta el riesgo explícitamente antes de proceder.

Coordinación de equipo

Múltiples miembros del equipo pueden trabajar en el mismo Claude Project. Estas reglas evitan conflictos y garantizan que el proceso avance de manera coherente.

T1
PROGRESS.md como sincronizador
Antes de trabajar en cualquier sesión, Claude lee el PROGRESS.md. Cualquier cambio al estado del proyecto se commitea inmediatamente. El equipo siempre sabe dónde están mirando el PROGRESS.md.
T2
Una persona activa a la vez
Solo un miembro del equipo trabaja activamente en el proceso en un momento dado. Si el PROGRESS.md fue actualizado hace menos de 2 horas, Claude avisa antes de hacer cambios y sugiere confirmar con el equipo.
T3
Commits inmediatos
Todo cambio al artifact, PROGRESS.md o CLAUDE.md se commitea inmediatamente al finalizar la sesión o el paso. Nunca se trabaja con cambios no commiteados entre sesiones del equipo.
T4
El artifact es la fuente de verdad
Si hay contradicción entre lo que alguien recuerda y lo que está en el artifact o PROGRESS.md, gana el documento. Claude señala la contradicción antes de proceder.
Guía al usuario

Claude guía activamente al usuario en cada momento del proceso. El usuario nunca debe preguntarse "¿qué hago ahora?" — Claude siempre tiene una respuesta clara y contextual. Estos son los tres momentos clave de guía.

Inicio de sesión Status Check

Al iniciar cualquier sesión nueva, Claude lee el PROGRESS.md del proyecto y presenta un resumen de estado: dónde están, qué se hizo la última vez, y cuál es la próxima acción exacta. El usuario no necesita recordar nada ni ponerse al día manualmente.

"📍 Status Check — [Nombre del Proyecto]
Estás en Fase 3 · Análisis de Mercado.
La última sesión completamos el análisis de competidores directos.

Criterios pendientes para cerrar esta fase:
☐ Análisis de tamaño de mercado (TAM/SAM/SOM)
☐ Oportunidades y riesgos
☐ Estrategia de posicionamiento

Siguiente paso: Análisis de tamaño de mercado. ¿Empezamos?"
Durante la sesión Orientación contextual

Si el usuario hace una pregunta fuera del contexto actual, Claude la responde pero recuerda en qué parte del proceso están y qué queda pendiente. Si el usuario parece perdido o no sabe cómo continuar, Claude toma la iniciativa y propone el siguiente paso concreto.

"Respondí tu pregunta sobre el stack. Recuerda que aún estamos en Fase 3 y nos falta completar el análisis de mercado antes de llegar a la planificación técnica donde eso aplica. ¿Continuamos con el TAM/SAM/SOM?"
Cierre de sesión o fase Bloque "¿Qué sigue?"

Al cerrar una fase o cuando el usuario indica que termina la sesión, Claude actualiza el PROGRESS.md, hace commit a GitHub, y presenta un bloque claro con los próximos 1–3 pasos concretos para la siguiente sesión.

"✅ Fase 3 completada. He actualizado el PROGRESS.md y el artifact.

¿Qué sigue?
1. Próxima sesión: Fase 4 · Planificación Técnica en Cowork con Opus 4.8
2. Qué traer: No necesitas nada — tengo todo el contexto en el artifact
3. Tiempo estimado: 1–2 sesiones para definir stack y arquitectura completa"
Proyectos existentes — Onboarding & Backfill

Para proyectos que ya están iniciados o en producción, existe una Fase 0 de Auditoría que antecede al framework estándar. Claude evalúa el estado real del proyecto, mapea lo que existe a las fases del framework, identifica gaps, y genera un Backfill Plan con acciones obligatorias y opcionales.

Fase 0 · Project Audit — 4 pasos
01
Dump de contexto
El usuario comparte todo lo que existe: código, docs, decisiones, wireframes, conversaciones previas, cualquier contexto disponible. Sin filtro.
02
Mapeo al framework
Claude analiza todo lo recibido y mapea cada elemento a la fase del framework que corresponde. Identifica qué está cubierto, qué está incompleto, y qué está ausente.
03
Backfill Plan
Claude genera el plan de backfill con dos categorías: gaps obligatorios (bloquean el avance) y gaps opcionales (se pueden aceptar con override documentado).
04
Inicialización
Una vez aprobado el Backfill Plan, Claude inicializa el framework: crea el PROGRESS.md, genera el artifact con lo que existe, y configura el repo en GitHub.
Ejemplo de Backfill Plan
Fase Estado Acción requerida
1 · Investigación Completo Existía contexto suficiente. Claude lo documenta en el artifact.
2 · Discovery Completo Visión y scope ya definidos. Claude los extrae y formaliza.
3 · Análisis Mercado Obligatorio No existe análisis formal. Debe completarse antes de avanzar — sin esto no hay base estratégica documentada.
4 · Plan. Técnica Opcional El stack está definido informalmente. Se puede aceptar con override documentado o formalizarlo — recomendado si el equipo va a cambiar.
5 · Diseño / UX Obligatorio No hay design system documentado. Debe crearse aunque sea retrospectivamente para que Claude Code trabaje con referencias claras.
6–7 · MVP / Dev Activo En curso — el código existente se integra al flujo de Fases 6–7, usando el design system retrospectivo como referencia.
